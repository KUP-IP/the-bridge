# Sparkle Update-Failure Triage (PKT-932)

**Triage date:** 2026-06-24  
**Branch:** pkt-932-sparkle-triage  
**Build status:** green (`make build` passes)

---

## Summary

The Sparkle update pipeline was analyzed end-to-end. The investigation covered:

1. Appcast XML vs Info.plist version parity
2. EdDSA pubkey chain (SUPublicEDKey in Info.plist vs appcast `edSignature`)
3. Enclosure DMG deliverability (URL reachability + Content-Length match)
4. In-app updater wiring (SPUStandardUpdaterController, UpdaterLogger delegate, checkForUpdates routing)
5. Staged-update resource-bundle integrity guards (StagedUpdateValidator, MenuBarIconResolver)
6. AdvancedSection "Check for updates" button delegate resolution

**Result:** the update channel is HEALTHY for v3.8.2. One in-app fragility was found and fixed (AdvancedSection delegate cast). All other findings are informational or require a live GUI + network smoke to verify.

## What Was Inspected

| `CFBundleVersion` | `61` |

The installed `/Applications/The Bridge.app` carries the same `SUPublicEDKey` — the pubkey chain is intact.

### 2. Appcast XML

Live feed (`https://raw.githubusercontent.com/KUP-IP/the-bridge/main/appcast.xml`) returns HTTP 200 and valid XML. The newest item:

- `sparkle:shortVersionString`: `3.8.2`
- `sparkle:version`: `61`
- enclosure URL: `https://github.com/KUP-IP/the-bridge/releases/download/v3.8.2/the-bridge-v3.8.2.dmg`
- declared `length`: `18614971`

Appcast version matches Info.plist exactly. No version parity issue.

### 3. Enclosure DMG deliverability

The DMG URL returns HTTP 302 → 200 (GitHub release redirect). The served `Content-Length` is `18614971`, which matches the appcast declared `length=`. **Deliverability is confirmed.** The v3.6.0 "enclosure not published" scenario (documented in `docs/release/sparkle-troubleshooting.md`) does NOT apply here.

### 4. EdDSA signature chain

The appcast `sparkle:edSignature` (`DcjA8rVtx1wZstvQFKhAOHcFRPYW+DgEFhjE34QP7I9e8snu0nHZGY78ZAHUN3qM9H0d8ky+RTeLXdm6EfO0Dg==`) was generated by the CI `release.yml` workflow using the `SPARKLE_ED_PRIVATE_KEY` secret. The public half is `AYjKcJUCz34FoCoeCEc17j7pd3enjqd2ttzDEV6F6fg=` (in Info.plist + installed app). Sparkle verifies the signature against `SUPublicEDKey` at download time.

**Headless verification limit:** confirming the Ed25519 signature cryptographically would require running `generate_appcast --verify` or a manual download + `sign_update` check — not possible in a headless triage. The correct tool for this is `make verify-sparkle-feed` (runs `check_appcast_version.sh` + `verify_sparkle_feed.sh`) after publishing a release. The pubkey in both the installed app and the repo Info.plist matches, so the chain is structurally sound.

### 5. In-app updater wiring (AppDelegate)

`SPUStandardUpdaterController` is initialized in `AppDelegate.init()` with:

- `startingUpdater: !runningInTestProcess` — correctly disarmed in the test harness to prevent `NSAlert.runModal` deadlock.
- `updaterDelegate: UpdaterLogger` — the `UpdaterLogger` class (PKT-932, added in a prior commit) provides `SPUUpdaterDelegate` conformance that logs every check outcome (found, not-found, aborted) to the unified log. Previously this was `nil`, which caused silent failures.
- `httpHeaders: ["Cache-Control": "no-cache"]` — prevents stale NSURLSession cache from masking a newly-published feed.

The `checkForUpdates()` method on `AppDelegate`:
- Flips activation policy to `.regular` + activates so Sparkle's update window can surface as a focused window.
- Logs `canCheckForUpdates`, `sessionInProgress`, and `feedURL` to the unified log.
- Surfaces an `NSAlert` when `canCheckForUpdates == false` (either a session is already in progress, or the updater is not yet ready) — replacing the historical silent no-op.

All of this was shipped in a prior commit (`UpdaterLogger` class). **The updater wiring is correct and complete.**

### 6. AdvancedSection "Check for updates" button — fragile delegate cast (FIXED)

**Finding:** The "Check for updates" button in `TheBridge/UI/Sections/AdvancedSection.swift` called:

```swift
(NSApp.delegate as? AppDelegate)?.checkForUpdates()
```

Under `@NSApplicationDelegateAdaptor`, `NSApp.delegate` may return the SwiftUI wrapper object rather than the concrete `AppDelegate` instance, causing the `as? AppDelegate` cast to silently fail. This is the same fragility that was previously documented and patched for `bridge_settings_navigate` (PKT-1005). If the cast fails, the button tap is a no-op — no update check fires, no error is shown.

**Fix applied:** The button now uses `AppDelegate.shared` (the identity-correct stable self-reference set in `applicationDidFinishLaunching`) with the cast as fallback:

```swift
(AppDelegate.shared ?? (NSApp.delegate as? AppDelegate))?.checkForUpdates()
```

This matches the pattern already used in `CommandBridge.swift` and `BridgeAutomationModule.swift`.

### 7. Staged-update bundle corruption guards

Already shipped (prior commits). Key components:

- `MenuBarIconResolver` — non-trapping icon resolution via `Bundle(path:)` candidate probing + SF Symbol fallback. Ensures the app always boots even with a corrupt SPM resource bundle.
- `StagedUpdateValidator` — pure predicate that detects empty-husk `.bundle` directories (the 2026-06-05 incident signature). Wired into `UpdaterLogger.shouldProceedWithUpdate` to abort an update if the *currently-running* app's resource bundle is already corrupt.
- `SparkleResilienceTests` — 14 regression tests covering both guards. All pass (verified in this triage run).

The **Sparkle API limitation** is documented: `shouldProceedWithUpdate` fires before download/extract (the staged bundle doesn't exist yet), so there is no Sparkle hook that can validate and reject the staged bundle post-extract. The definitive protection is graceful degradation at the load site, not the pre-swap guard.

| # | Issue | Root cause | Fixed? |

| 1 | "Check for Updates" was a silent no-op (historical, pre-PKT-932) | `updaterDelegate: nil` — no error logging; no alert for `canCheckForUpdates == false` | Yes (prior commit, `UpdaterLogger`) |

## Manual Step Required (Operator)

A live end-to-end update smoke cannot be performed headlessly. The following confirms the full update applies:

1. Install a prior signed build (e.g. v3.8.1) into `/Applications`.
2. Launch The Bridge; open Settings → Advanced.
3. Tap "Check for updates".
4. Confirm:
   - The unified log (`log stream --predicate 'process == "TheBridge"'`) shows `[Bridge][Updater] found valid update: 3.8.2` (or `no update found` if already at 3.8.2).
   - A Sparkle update sheet appears (if an update is found).
   - The update downloads, installs, and relaunches at the newer version.
5. Confirm the menu-bar icon is present after relaunch (not the SF Symbol fallback).

The automated `make verify-sparkle-feed` script (`scripts/verify_sparkle_feed.sh`) is the recommended post-release check and covers steps 1–3 of the deliverability chain.
