## 1 · `TheBridge/Server/` vs `TheBridge/Modules/` paths

Prior packet narratives (PKT-746 fail-fast, Reflow #14 PM Directive) inconsistently referenced `TheBridge/Server/` as the home for module sources. The actual on-disk layout is **flat and split-by-concern**:

- `TheBridge/Server/` — transport + routing only. Contains `ServerManager.swift` (router lifecycle + module registration), `ToolRouter.swift` (registration store), `SSETransport.swift`, `MCPHTTPValidation.swift`. Module *implementations* do not live here.
- `TheBridge/Modules/` — flat directory of all module sources (one Swift file per logical module, plus shared runtimes). Per Decision #29 (Reflow #13), this flat layout is **canonical**; the earlier “registered under dev/” phrasing from PKT-738 (0.1 scaffold) referred to the *module-string* `"dev"` namespace, not a directory path.

**Implication for runners:** `PlaywrightModule.swift`, `VitestModule.swift`, `LighthouseModule.swift`, plus the shared `RunnerSupport.swift` (capability probe + tool impl) and `RunnerParsers.swift` (typed JSON envelope), all live directly in `TheBridge/Modules/`. They are registered with the router from `TheBridge/Server/ServerManager.swift`.

## 2 · Per-runner module layout

The 3.2a (PKT-781) + 3.2b (PKT-782) chain delivered **three separate runner modules** sharing two helper files, not a single monolithic `RunnersModule`. Final shape:

| File | Purpose | PKT |
|---|---|---|
| `PlaywrightModule.swift` | Registers `playwright_run` tool. Defines `moduleName = "dev"`, `toolName = "playwright_run"`, `runnerName = "playwright"`. | 781 (3.2a) |
| `VitestModule.swift` | Registers `vitest_run` tool. Same pattern; `runnerName = "vitest"`. Capability probe expected to short-circuit `capability_missing` on this host (no global vitest install). | 781 (3.2a) |

This split keeps each runner’s registration + arg schema isolated for evolution while sharing the capability-probe and bg_process dispatch logic. PKT-3.2c (e2e) consumes the live registered tools; it does not import `RunnerParsers` directly — parsers are invoked from inside each runner’s handler after `bg_process` completion.

## 3 · `module="dev"` tier `.request` registration semantics

All three runner modules declare:

```swift
public static let moduleName = "dev"
```

and register their tool with `tier: .request`. Two consequences worth flagging for downstream consumers:

1. **`tools_list({module: "dev"})` is the canonical discovery surface.** A filter on `module = "dev"` returns every dev-namespace tool from every backing Swift module, including the runner trio. Filtering by Swift type name (e.g. `PlaywrightModule`) is not supported and would miss tools.
2. **`.request` tier means SecurityGate Request-tier handling.** Per the SecurityGate cascade, `.request` tools surface a confirmation prompt unless covered by an Always-Allow scope. They are *not* `.open` (read-only) and *not* `.elevated` (admin-only). Test harnesses and HITL runs (PKT-3.2c) should expect approval prompts on first invocation per (cwd, runner) tuple.
