## 1. Surfacing model — *when does memory reach the agent?*

Today: agents get memory only by **explicitly** calling `memory_recall` or reading `bridge://memory`. Nothing is pushed.

**Questions:**
- **Q1a.** Should the handshake **auto-inject** the salient memory slice into `initialize.instructions` (every connecting agent sees recent memory with zero tool calls), or stay **opt-in** (resource read only)? The composer is sync + byte-deterministic today; auto-inject makes it async/variable — is that acceptable, or should it be a separate post-handshake push?
- **Q1b.** If auto-inject: **flag-gated default ON or OFF?** Global, or per-client (the `ClientOverlayStore` precedent allows per-client)?
- **Q1c.** **Memory-rides-routing** — when an agent fetches a skill via `fetch_skill(parent, intent)`, should scoped memory for that intent ride along automatically? (Named in the deferred plan; needs a yes/no and a scoping rule.)
- **Q1d.** What's the **token budget** for any auto-surfaced slice, and what happens when memory exceeds it (truncate by salience? summarize?)?

> **Suggested default to react to:** opt-in resource stays the baseline; add a **per-client flag-gated auto-inject defaulting OFF**, capped at a fixed token budget, truncating by salience. Memory-rides-routing deferred to a later patch unless you want it in 3.7.7.

## 2. Lifecycle / TTL — *how does memory age, demote, and die?*

Today: entries never expire on their own. `forget` is a manual soft-tombstone. Recency only **down-weights** salience; it never removes. There is no consolidation Job.

**Questions:**
- **Q2a.** Do we want **tiered demote-not-delete** (the architecture you locked: hot → warm → cold, demoted out of default recall but retrievable), and if so what are the tier boundaries (age? salience floor? use-count floor?)?
- **Q2b.** **TTL semantics:** should `memory_remember` accept an explicit TTL/expiry, and is there a *default* TTL by `type` (e.g. `reference` shorter, `decision` indefinite)? Or no automatic TTL at all — demotion only?
- **Q2c.** **Invalidation:** when a superseding fact arrives, the old row is already tombstoned. But what about facts that go stale without an explicit replacement (e.g. "currently on branch X")? Manual-only, or a staleness signal?
- **Q2d.** **Idle consolidation Job** — the deferred plan names a background pass that merges/summarizes/demotes. In scope for 3.7.7? If yes: cadence, and merge policy (it would extend the existing dedup, which is currently same-scope+entity only).
- **Q2e.** **Startup catch-up** — should a demote/consolidation sweep run on app launch (like the v3.7.6 weekly credential auto-validate `>7d` on-launch pattern), given Jobs infra can't host an internal Swift call?

> **Suggested default to react to:** ship **demote-not-delete tiers + optional explicit TTL** in 3.7.7; default TTL by type = none (indefinite) except `reference`; run consolidation as an **on-launch catch-up sweep** (not a Jobs entry), cadence-throttled. Automatic staleness detection deferred.

## 3. Dedup across keepers / sources — *whose memory is it?*

Today: dedup keys on `(scope, entity, contentHash | Jaccard)`. `source` is a free-text label that defaults to `"bridge-mcp"` — it does **not** participate in dedup, and the live MCP client name is **not yet threaded** into the handler (see `MemoryModule` lines 16–22 — `source` is the clean seam for this).

**Questions:**
- **Q3a.** Should dedup remain **source-agnostic** (Claude Code, Cursor, Raycast, ChatGPT all collapse into one shared memory per scope+entity) — i.e. one operator brain — or **source-partitioned** (each client has its own memory lane)?
- **Q3b.** Should we **thread the real MCP client name** into `source` now (the deferred seam), so provenance is truthful regardless of the Q3a answer?
- **Q3c.** **Conflict policy** when two sources assert contradicting facts in the same scope+entity below the Jaccard threshold (so not auto-superseded): keep both? surface both at recall with provenance? last-writer-wins? operator-arbitrated?
- **Q3d.** Does `scope` need a **reserved taxonomy** (people | project | mac | time | skill | global) or stay open-ended? Cross-keeper dedup only works if scopes are consistent across clients.

> **Suggested default to react to:** **source-agnostic shared memory** (one operator brain — matches the KEEP OS trajectory), but **thread the real client name into `source` for provenance**. Contradictions below threshold: **keep both, surface both with provenance** at recall. Keep scope open-ended but document the canonical set.

## 4. UI surface — *how does the operator see and govern memory?*

Today: no UI. Memory is invisible in the app; only agents touch it via tools/resource.

**Questions:**
- **Q4a.** Does 3.7.7 add a **memory-health / review UI** (a Settings section listing entries, with pin/forget/edit)? It is named in the deferred plan.
- **Q4b.** If yes: read-only inspector first, or full **CRUD** (operator can author/edit/delete memories directly)? Hard-delete from UI, or soft-tombstone only (consistent with the tool path)?
- **Q4c.** Where does it live — its own left-nav section (like Credentials/Standing Orders), or folded into an existing one?
- **Q4d.** Any **notification/telemetry** surface (the `DeliveryLog` precedent) — e.g. "N memories surfaced this session," memory freshness, consolidation runs?
- **Q4e.** Theme: must follow the v3.7.6 system-tethered adaptive tokens (carbon/titanium) — confirmed constraint, not a question, but flag if the inspector needs new tokens.

> **Suggested default to react to:** ship a **read-only memory inspector** in its own left-nav section in 3.7.7 (pin/forget actions only, soft-tombstone consistent with tools); defer full authoring CRUD; add a lightweight "memories surfaced" line to the existing Delivery-audit telemetry rather than a new surface.

## 5. Storage / sync — *where does memory live, and does it travel?*

Today: a **single local SQLite file** at `~/.config/notion-bridge/memory.sqlite` (relocates with `BRIDGE_CONFIG_PATH`). No sync, no backup, no cloud.

**Questions:**
- **Q5a.** Stays **local-only** in 3.7.7, or is **sync** in scope (multi-Mac, or cloud-backed)? The product-strategy framing ("trusted neutral broker that holds your state") suggests sync is eventually core — but is it *this* patch?
- **Q5b.** If sync: what backbone — the existing WorkOS/cloud-access path, Notion as the store-of-record, or a dedicated sync? (Each has very different privacy posture; see `PRIVACY.md`.)
- **Q5c.** **Backup / export** — should `memory_*` gain an export/import (the snippets module has `snippets_export/import` as precedent) even if sync is deferred?
- **Q5d.** **Encryption at rest** — credentials use the Keychain + Touch-ID gate; memory is currently plaintext SQLite. Is operator-personal memory sensitive enough to warrant envelope encryption, or is local-disk trust sufficient?

> **Suggested default to react to:** **local-only in 3.7.7**; add **export/import** (snippets precedent) as the seam toward future sync; **defer cloud sync and at-rest encryption** to a later patch with its own privacy review. Confirm plaintext-local is acceptable for the operator's threat model.

## 6. ⚑ Load-bearing: *what does "Unified" unify?* (answer this first)

There are currently **two independent memory systems** in this environment:

1. **The Bridge store** — SQLite/FTS5, `memory_remember`/`memory_recall`, `bridge://memory`. Agent-written, salience-ranked, dedup'd.
2. **The Claude Code auto-memory** — `MEMORY.md` index + per-topic markdown files under `~/.claude/projects/-Users-keepup-Developer-the-bridge/memory/` (currently 8 topic files: contacts-sync, business-offerings, balance-venture, notion-markdown-api, documentation-depth, v3.7.2-resurface, handshake-program, v3.7.6-page-polish). Harness-written, human-readable, version-light.

These **do not talk to each other.** A fact remembered in one is invisible to the other.

**The question that gates the whole packet:**
- **Q6.** Does v3.7.7 **reconcile these two systems into one** (the literal reading of "Unified Memory" — e.g. the Bridge store becomes the SSOT and the MEMORY.md files are ingested/mirrored, or vice-versa), or does "Unified" mean **only "one Bridge memory across all MCP clients"** and the Claude Code `MEMORY.md` system stays a separate, parallel track?

Every answer above (surfacing budget, dedup keys, UI scope, storage backbone) changes depending on whether MEMORY.md is in scope. **This is the upstream decision.**

> **Suggested default to react to:** scope v3.7.7 to **"one Bridge memory across all MCP clients"** (systems stay separate; MEMORY.md is the harness's lane). Treat Bridge↔MEMORY.md ingestion as a **named future packet**, not 3.7.7 — unless you intend the bigger unification now, in which case it reshapes the whole patch and the decisions above must be re-scoped against it.
