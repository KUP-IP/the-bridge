## Honest carve-outs (deferred to v3.7)

### v3.7·A — Skills cache pipeline
- **What is missing:** the on-disk skills cache that `RoutingIndex` would
  populate from. Currently `StandingOrdersSection.cachedRoutingSkills()`
  returns `[]` (TODO at line 244).
- **Why deferred:** the composer + UI render correctly without it (the
  "None registered yet" path). Wiring the cache writer is a distinct
  packet (sync from Notion → on-disk cache → reader → routing index)
  whose scope crosses the SkillsManager/NotionClient boundary.
- **Follow-on packet:** "v3.7 — SkillsCacheReader/Writer pipeline".

### v3.7·B — `standing_orders_*` MCP tools
- **What is missing:** the `standing_orders_read` and `standing_orders_write`
  MCP tools that the Store and Composer were designed to back. They are
  referenced in source comments
  (`StandingOrdersStore.swift:6`, `StandingOrdersComposer.swift:4`) but not
  registered with the ToolRouter.
- **Why deferred:** the in-Settings UI surface already exposes read/write
  via the SwiftUI editor. The MCP tool surface is for programmatic access
  by other agents; landing it cleanly requires tool annotations, security
  tier assignment (almost certainly `request` for write, `open` for read),
  schema definitions, and a complete test file. Out of audit-polish scope.
- **Follow-on packet:** "v3.7 — standing_orders MCP tool surface".

## What MCP clients see today

A connecting MCP client receives the composed `orders.md` via the existing
Standing Orders surface (the StandingOrdersSection's composed preview is
the same bytes). Per-client overlays apply if a sibling
`orders.<client-prefix>.md` exists in
`~/Library/Application Support/The Bridge/standing-orders/`. The Routing
Skills trailer is appended automatically (or suppressed when an inline
`## Routing skills` section is present in the curated body).
