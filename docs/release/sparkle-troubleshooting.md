# Sparkle Update Troubleshooting (PKT-932)

How to diagnose and prevent the "Update failure" dialog users hit when checking
for updates.

## TL;DR — the most common cause

The appcast (`appcast.xml`, served at `SUFeedURL`) advertises a version, but the
**enclosure DMG it points at is missing or the wrong size**. Sparkle sees a valid
newer version, starts the download, and fails — surfacing the failure dialog.

A reachable *feed* does **not** prove an update can complete. The *enclosure* has
to be published too. This is exactly what happened in the v3.6.0 → v3.7 window:
the appcast was bumped to 3.6.0 before the GitHub release + DMG asset existed, so
the enclosure URL 404'd until the release was cut (2026-05-27).

## One-command check

```sh
make verify-sparkle-feed         # = check-appcast + verify_sparkle_feed.sh
# or directly:
./scripts/verify_sparkle_feed.sh Info.plist
```

This now verifies, in order:

1. `check_appcast_version.sh` — appcast `sparkle:shortVersionString` / `sparkle:version`
   match `Info.plist` `CFBundleShortVersionString` / `CFBundleVersion`.
2. `SUFeedURL` returns HTTP 200 and is XML (the feed is reachable).
3. **PKT-932:** the appcast `<enclosure>` DMG URL returns HTTP 200 **and** the served
   `Content-Length` equals the appcast's declared `length=`.

Run it **after** publishing a release (it hits the live feed).

## Failure modes & fixes

| Symptom | Likely cause | Fix |
|---|---|---|
| Enclosure URL → HTTP 404 | Release/DMG asset not published (appcast bumped ahead of the release) | `gh release create vX.Y.Z` with the signed DMG attached, or revert the appcast to the last published version |
| Enclosure length mismatch | DMG re-uploaded after the appcast was signed | Regenerate the appcast from the published DMG: `make appcast`, recommit |
| Signature error in Sparkle log | `sparkle:edSignature` doesn't match the DMG, or the DMG was signed with a key whose public half isn't the embedded `SUPublicEDKey` | Regenerate the appcast (`make appcast`) from the exact published DMG with the release EdDSA key; confirm `SUPublicEDKey` in `Info.plist` matches the signer |
| "You're up to date" (no dialog) | appcast `sparkle:version` ≤ installed `CFBundleVersion` | Not a failure — bump + re-cut if a newer build exists |
| Feed → 404 | Repo private / wrong `SUFeedURL` | Repo must be public (raw GitHub URL) or host the appcast on a public HTTPS URL |

## Manual end-to-end verification (operator)

Automated checks confirm the *artifacts* are deliverable. To confirm the *update
applies*, and to clear the PKT-932 DoD:

1. Install a prior build (e.g. v3.6.0) into `/Applications`.
2. Force an update check (menu → Check for Updates, or relaunch).
3. Confirm the update downloads, installs, and relaunches at the newer version.
4. **Grandfather safety (release-blocker):** install a v3.4.x build with the
   migration sentinel, update to current, and confirm the licensing surface
   treats the user as `.grandfathered` — no trial countdown. See
   `LicenseManager` + `PathMigration` (`.bridge-migration-v3.5-complete`).
