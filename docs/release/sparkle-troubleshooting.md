# Sparkle Update Troubleshooting (PKT-932)

How to diagnose and prevent the "Update failure" dialog users hit when checking
for updates.

## TL;DR — the most common cause

The appcast (`appcast.xml`, served at `SUFeedURL`) advertises a version, but the
**enclosure DMG it points at is missing or the wrong size**. Sparkle sees a valid
newer version, starts the download, and fails — surfacing the failure dialog.

A reachable *feed* does **not** prove an update can complete. The *enclosure* has
to be published too. This is exactly what happened in the v3.6.0 → v3.7 window:
the appcast was bumped to 3.6.0 before the GitHub release + DMG asset existed, so
the enclosure URL 404'd until the release was cut (2026-05-27).

## One-command check

```sh
make verify-sparkle-feed         # = check-appcast + verify_sparkle_feed.sh
# or directly:
./scripts/verify_sparkle_feed.sh Info.plist
```

This now verifies, in order:

1. `check_appcast_version.sh` — appcast `sparkle:shortVersionString` / `sparkle:version`
   match `Info.plist` `CFBundleShortVersionString` / `CFBundleVersion`.
2. `SUFeedURL` returns HTTP 200 and is XML (the feed is reachable).
3. **PKT-932:** the appcast `<enclosure>` DMG URL returns HTTP 200 **and** the served
   `Content-Length` equals the appcast's declared `length=`.

Run it **after** publishing a release (it hits the live feed).

## Failure modes & fixes

| Symptom | Likely cause | Fix |
|---|---|---|
| Enclosure URL → HTTP 404 | Release/DMG asset not published (appcast bumped ahead of the release) | `gh release create vX.Y.Z` with the signed DMG attached, or revert the appcast to the last published version |
| Enclosure length mismatch | DMG re-uploaded after the appcast was signed | Regenerate the appcast from the published DMG: `make appcast`, recommit |
| Signature error in Sparkle log | `sparkle:edSignature` doesn't match the DMG, or the DMG was signed with a key whose public half isn't the embedded `SUPublicEDKey` | Regenerate the appcast (`make appcast`) from the exact published DMG with the release EdDSA key; confirm `SUPublicEDKey` in `Info.plist` matches the signer |
| "You're up to date" (no dialog) | appcast `sparkle:version` ≤ installed `CFBundleVersion` | Not a failure — bump + re-cut if a newer build exists |
| Feed → 404 | Repo private / wrong `SUFeedURL` | Repo must be public (raw GitHub URL) or host the appcast on a public HTTPS URL |

## Manual end-to-end verification (operator)

Automated checks confirm the *artifacts* are deliverable. To confirm the *update
applies*, and to clear the PKT-932 DoD:

1. Install a prior build (e.g. v3.6.0) into `/Applications`.
2. Force an update check (menu → Check for Updates, or relaunch).
3. Confirm the update downloads, installs, and relaunches at the newer version.
4. **Grandfather safety (release-blocker):** install a v3.4.x build with the
   migration sentinel, update to current, and confirm the licensing surface
   treats the user as `.grandfathered` — no trial countdown. See
   `LicenseManager` + `PathMigration` (`.bridge-migration-v3.5-complete`).

## Key references

- Feed: `Info.plist` → `SUFeedURL`, `SUPublicEDKey`
- Appcast: `appcast.xml` (generated by `make appcast` via Sparkle `generate_appcast`)
- Make targets: `check-appcast`, `verify-sparkle-feed`, `check-update-flow`
- Scripts: `scripts/check_appcast_version.sh`, `scripts/verify_sparkle_feed.sh`

# Staged-update SPM-bundle corruption → menu-bar crash-loop (2026-06-05)

A SECOND, more dangerous failure mode — distinct from the "update failure"
dialog above. Here the update *applied*, but left the app in a
**bootable-but-crash-looping** state that no dialog can recover from.

## Root cause

The app's menu-bar icon (and bundled skills) live in the SwiftPM **resource
bundles** `TheBridge_TheBridge.bundle` (executable target) and
`TheBridge_TheBridgeLib.bundle` (library target), copied into
`The Bridge.app/Contents/Resources/` by the Makefile `app` target.

On 2026-06-05 a Sparkle **staged-update swap** raced/partially-applied and left
`TheBridge_TheBridge.bundle` on disk **without a `Contents/` directory** —
a structurally-corrupt bundle. The old code reached the icon through SwiftPM's
generated `Bundle.module` accessor, which is **not** a normal optional lookup:
SwiftPM synthesizes it to call `Swift.fatalError` ("could not load resource
bundle") when neither candidate path loads. So:

```
loadMenuBarIcon()  →  Bundle.module  (static-init)  →  Bundle(path:) == nil
                   →  Swift.fatalError  →  EXC_BREAKPOINT / SIGTRAP
```

…on **every launch**. The app booted far enough to crash, so the watchdog/
relaunch loop just re-crashed. Recovery required a manual clear-staging +
reinstall.

## The guards (this fix — `feat/fb-sparkle-atomic-install`)

### 1. Graceful degradation (the crash-loop killer — definitive)

`TheBridge/App/MenuBarIconResolver.swift` resolves the menu-bar icon WITHOUT
ever touching the trapping `Bundle.module` accessor:

- It probes candidate resource-bundle paths via `Bundle(path:)`, which **returns
  `nil`** for a missing/corrupt bundle (never traps), then `Bundle.main` (the
  top-level `MenuBarIcon.png` copy).
- If nothing loads, it falls back to a **system SF Symbol**
  (`rectangle.connected.to.line.below`) and logs the degradation.

`loadMenuBarIcon()` (in `TheBridgeApp.swift`) now returns a non-optional
`NSImage` from the resolver, so the menu-bar item is **always** rendered and the
app **always boots** — worst case it shows the SF Symbol glyph plus a unified-log
line pointing here. `FilesystemSkillIndex.defaultBundledDirectory()` was likewise
moved off `Bundle.module` to a non-trapping `Bundle(path:)` lookup (bundled
skills are optional; a broken bundle now degrades to "no bundled skills" instead
of trapping the server-bootstrap path).

### 2. Pre-swap validation (best-effort — Sparkle API limited)

`TheBridge/App/StagedUpdateValidator.swift` is a pure, non-trapping predicate
that flags the incident signature (a `.bundle` that exists but has no
`Contents/`, or is unloadable). It is wired into the `SPUUpdaterDelegate`
(`UpdaterLogger` in `AppDelegate.swift`):

- **`updater(_:shouldProceedWithUpdate:updateCheck:)`** — Sparkle's *only*
  abort-capable hook. It **throws** (aborting the update) if the
  **currently-running** app's resource bundle is already corrupt, so an
  install-on-top from a poisoned base can't perpetuate the crash-loop; the
  operator is told to reinstall first.
- **`didExtractUpdate` / `willInstallUpdate`** — VOID notification hooks; they
  log the install transition so a future corruption is correlatable in the log.

**Why we can't validate the *staged* bundle and abort the swap:** Sparkle's
`shouldProceedWithUpdate` runs **before** download/extract, so the staged app
doesn't exist yet to inspect. The post-extract hooks (`didExtractUpdate`,
`willInstallUpdate`) have **no return value** and cannot abort. The atomic swap
itself is done by Sparkle's **sandboxed Installer XPC service**, to which the
running app has no reliable filesystem path. There is therefore no clean Sparkle
seam to validate the staged bundle and reject the swap post-extract — so we rely
on **(1) graceful degradation** (guarantees the app always boots regardless) plus
the **install-copy hardening** (PR #31) plus this **running-app integrity gate**.

### 3. Install hardening (PR #31 — already in the Makefile)

The `install` / `install-copy` targets quit the running app, **clear pending
Sparkle staging** before writing, and **verify** version + SPM-bundle presence
after (`PREINSTALL_SAFETY` / `VERIFY_INSTALL` in the `Makefile`). This prevents a
local `make install` from racing Sparkle and producing the corrupt bundle.

## Manual recovery (operator)

If The Bridge crash-loops at launch after an update (no menu-bar item, or it
appears then vanishes), recover by clearing Sparkle staging and reinstalling:

```sh
# 1. Quit any running instance
osascript -e 'tell application "The Bridge" to quit' 2>/dev/null || true
pkill -f "The Bridge.app/Contents/MacOS/TheBridge" 2>/dev/null || true

# 2. Clear pending Sparkle staged update (the corrupt swap source)
rm -rf "$HOME/Library/Caches/kup.solutions.notion-bridge/org.sparkle-project.Sparkle/Installation/"*
rm -rf "$HOME/Library/Caches/kup.solutions.notion-bridge/org.sparkle-project.Sparkle/PersistentDownloads/"*

# 3. Reinstall a known-good build (clears + re-copies the SPM resource bundle)
make install-copy
```

(`make install-copy` performs steps 1–2 automatically via `PREINSTALL_SAFETY`,
then re-copies and verifies the bundle via `VERIFY_INSTALL`.)

With the graceful-degradation guard shipped, an app on this version will **boot
with an SF Symbol icon** even if the bundle is corrupt — so recovery becomes
"reinstall at your convenience" rather than an emergency.

## Operator residual (cannot be unit-tested)

A live **cross-version end-to-end update** (install a prior signed build, let
Sparkle stage + swap to current, confirm it relaunches healthy AND that an
artificially-corrupted resource bundle degrades to the SF Symbol icon instead of
crash-looping) is an operator manual smoke — it requires real Sparkle staging +
the Installer XPC + a relaunch, none of which the headless test harness can
exercise. The regression suite covers the pure guard logic
(`SparkleResilienceTests`); the live swap is the residual.
